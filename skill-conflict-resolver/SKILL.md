# Skill 冲突解决器

当用户任务场景同时满足多个 skill 的触发场景时，帮助用户做出明智的选择。

## 调用时机

当检测到以下情况时自动触发：

- 用户请求的任务同时匹配多个 skill 的触发条件
- 全局 skill 和项目 skill 都适用于当前任务
- 用户明确表示不确定应该使用哪个 skill
- 任务场景存在多种可能的实现路径

## 核心原则

### 优先级规则

当多个 skill 同时满足触发条件时，按以下优先级处理：

1. **项目 skill 优先**：当全局 skill 和项目 skill 同时满足触发条件时，优先使用项目 skill
2. **用户选择优先**：当存在多个合适 skill 时，询问用户偏好
3. **上下文相关**：根据当前工作环境和任务上下文推荐最合适的 skill

### Skill 分类

| 类型 | 位置 | 优先级 | 说明 |
|------|------|--------|------|
| 项目 skill | `.trae/skills/` | 高 | 项目特定的工作流和规范 |
| 全局 skill | 全局配置目录 | 低 | 通用的工作流和规范 |

## 冲突检测流程

### 步骤 1：识别匹配的 skill

分析用户任务，识别所有匹配的 skill：

```markdown
任务：[用户任务描述]

匹配的 skill：
1. [skill-name-1] - [触发原因]
2. [skill-name-2] - [触发原因]
3. [skill-name-3] - [触发原因]
```

### 步骤 2：分类和排序

将匹配的 skill 按类型分类：

- **项目 skill**：位于 `.trae/skills/` 目录
- **全局 skill**：位于全局配置目录

按优先级排序：项目 skill > 全局 skill

### 步骤 3：生成对比信息

为每个匹配的 skill 生成简要对比：

| skill 名称 | 类型 | 核心功能 | 适用场景 | 推荐度 |
|------------|------|----------|----------|--------|
| [name] | 项目/全局 | [功能] | [场景] | ⭐⭐⭐ |

## 用户交互流程

### 标准提问格式

使用 `AskUserQuestion` 工具向用户提问，格式如下：

```markdown
检测到您的任务同时匹配以下 skill：

**选项 1：[skill-name-1]**（项目 skill - 推荐）
- 功能：[核心功能描述]
- 适用场景：[适用场景描述]
- 推荐理由：[推荐理由，如"项目特定工作流"或"更符合当前上下文"]

**选项 2：[skill-name-2]**（全局 skill）
- 功能：[核心功能描述]
- 适用场景：[适用场景描述]

**选项 3：[skill-name-3]**（全局 skill）
- 功能：[核心功能描述]
- 适用场景：[适用场景描述]

请选择您希望使用的 skill：
```

### 提问规则

1. **最多展示 4 个选项**：避免信息过载
2. **项目 skill 排在前面**：并标记为"推荐"
3. **提供清晰的对比信息**：功能、适用场景、推荐理由
4. **允许用户选择"其他"**：提供自定义输入选项

## 使用示例

### 示例 1：项目 skill 与全局 skill 冲突

**用户任务**：创建一个新功能并编写测试

**检测到的 skill**：
- `test-driven-development`（项目 skill）：TDD 工作流
- `brainstorming`（全局 skill）：功能设计

**交互过程**：

```
检测到您的任务同时匹配以下 skill：

**选项 1：test-driven-development**（项目 skill - 推荐）
- 功能：测试驱动开发工作流，先写测试再实现
- 适用场景：需要实现新功能或修复 bug 时
- 推荐理由：项目特定的 TDD 工作流，符合团队规范

**选项 2：brainstorming**（全局 skill）
- 功能：在实现前探索用户意图、需求和设计
- 适用场景：创建功能、构建组件、添加功能或修改行为前

请选择您希望使用的 skill：
```

### 示例 2：多个全局 skill 冲突

**用户任务**：修复一个 bug 并请求代码审查

**检测到的 skill**：
- `systematic-debugging`（全局 skill）：系统化调试
- `requesting-code-review`（全局 skill）：请求代码审查

**交互过程**：

```
检测到您的任务同时匹配以下 skill：

**选项 1：systematic-debugging**（推荐）
- 功能：系统化调试流程，定位和修复 bug
- 适用场景：遇到 bug、测试失败或异常行为时
- 推荐理由：先修复 bug，再进行代码审查

**选项 2：requesting-code-review**
- 功能：请求代码审查，验证工作成果
- 适用场景：完成任务、实现重要功能或合并前

请选择您希望使用的 skill：
```

### 示例 3：三个以上 skill 冲突

**用户任务**：在新分支上实现功能、编写测试并提交代码

**检测到的 skill**：
- `using-git-worktrees`（全局 skill）：创建隔离工作树
- `test-driven-development`（项目 skill）：TDD 工作流
- `chinese-commit-conventions`（项目 skill）：中文提交规范

**交互过程**：

```
检测到您的任务同时匹配以下 skill：

**选项 1：using-git-worktrees**（推荐先执行）
- 功能：创建隔离的 git 工作树
- 适用场景：开始功能开发或执行实现计划前
- 推荐理由：首先创建隔离环境，再进行开发

**选项 2：test-driven-development**（项目 skill）
- 功能：测试驱动开发工作流
- 适用场景：实现功能或修复 bug 时
- 推荐理由：项目特定的 TDD 工作流

**选项 3：chinese-commit-conventions**（项目 skill）
- 功能：中文 Git 提交规范
- 适用场景：提交代码时
- 推荐理由：项目特定的提交规范

**选项 4：按顺序执行所有 skill**
- 按推荐顺序依次执行：worktrees → TDD → commit

请选择您希望使用的 skill：
```

## 特殊场景处理

### 场景 1：用户选择"其他"

当用户选择"其他"并提供自定义输入时：

1. 分析用户输入的意图
2. 如果用户指定了特定 skill，直接使用该 skill
3. 如果用户描述了新的工作流，按用户描述执行
4. 记录用户偏好，用于未来参考

### 场景 2：自动推荐

当满足以下条件时，可以自动推荐而不询问：

- 只有一个项目 skill 匹配
- 项目 skill 明显更适合当前上下文
- 用户之前在类似场景中选择了相同的 skill

**自动推荐格式**：

```
检测到您的任务匹配 [skill-name]（项目 skill）。
该 skill 最适合当前任务，将自动使用。

如需选择其他 skill，请告诉我。
```

### 场景 3：skill 组合

当用户需要按顺序执行多个 skill 时：

1. 识别 skill 之间的依赖关系
2. 确定合理的执行顺序
3. 提供组合执行选项
4. 在每个 skill 执行完成后确认是否继续

## 对比信息生成规则

### 功能描述

从 skill 的 description 字段提取核心功能：

```markdown
功能：[从 description 提取的功能描述]
```

### 适用场景

从 skill 的 description 或 body 中提取触发条件：

```markdown
适用场景：[从 description 或 body 提取的触发条件]
```

### 推荐理由

根据以下规则生成推荐理由：

| 情况 | 推荐理由 |
|------|----------|
| 项目 skill | "项目特定工作流，符合团队规范" |
| 更具体的 skill | "更具体的工作流，更适合当前任务" |
| 先决条件 skill | "需要先完成此步骤" |
| 用户历史偏好 | "您之前在类似场景中选择了此 skill" |

## 记住

1. **项目 skill 优先**：当全局 skill 和项目 skill 同时满足触发条件时，优先使用项目 skill
2. **提供清晰对比**：帮助用户理解每个 skill 的功能和适用场景
3. **尊重用户选择**：用户的选择优先于自动推荐
4. **避免信息过载**：最多展示 4 个选项
5. **记录用户偏好**：用于未来类似场景的推荐
